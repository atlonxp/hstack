## AskUserQuestion Format

**ALWAYS follow this structure for every AskUserQuestion call:**
1. **Re-ground:** State the project, the current branch (use the `_BRANCH` value printed by the preamble — NOT any branch from conversation history or gitStatus), and the current plan/task. (1-2 sentences)
2. **Simplify:** Explain the problem in plain English a smart 16-year-old could follow. No raw function names, no internal jargon, no implementation details. Use concrete examples and analogies. Say what it DOES, not what it's called.
3. **Recommend:** `RECOMMENDATION: Choose [X] because [one-line reason]` — always prefer the complete option over shortcuts (see Completeness Principle). Include `Completeness: X/10` for each option. Calibration: 10 = complete implementation (all edge cases, full coverage), 7 = covers happy path but skips some edges, 3 = shortcut that defers significant work. If both options are 8+, pick the higher; if one is ≤5, flag it.
4. **Options:** Lettered options: `A) ... B) ... C) ...` — when an option involves effort, show both scales: `(human: ~X / CC: ~Y)`

Assume the user hasn't looked at this window in 20 minutes and doesn't have the code open. If you'd need to read the source to understand your own explanation, it's too complex.

Per-skill instructions may add additional formatting rules on top of this baseline.

## Completeness Principle — Boil the Lake

AI makes completeness near-free. Always recommend the complete option over shortcuts — the delta is minutes with CC+gstack. A "lake" (100% coverage, all edge cases) is boilable; an "ocean" (full rewrite, multi-quarter migration) is not. Boil lakes, flag oceans.

**Effort reference** — always show both scales:

| Task type | Human team | CC+gstack | Compression |
|-----------|-----------|-----------|-------------|
| Boilerplate | 2 days | 15 min | ~100x |
| Tests | 1 day | 15 min | ~50x |
| Feature | 1 week | 30 min | ~30x |
| Bug fix | 4 hours | 15 min | ~20x |

Include `Completeness: X/10` for each option (10=all edge cases, 7=happy path, 3=shortcut).

## Step 0: Detect platform and base branch

First, detect the git hosting platform from the remote URL:

```bash
git remote get-url origin 2>/dev/null
```

- If the URL contains "github.com" → platform is **GitHub**
- If the URL contains "gitlab" → platform is **GitLab**
- Otherwise, check CLI availability:
  - `gh auth status 2>/dev/null` succeeds → platform is **GitHub** (covers GitHub Enterprise)
  - `glab auth status 2>/dev/null` succeeds → platform is **GitLab** (covers self-hosted)
  - Neither → **unknown** (use git-native commands only)

Determine which branch this PR/MR targets, or the repo's default branch if no
PR/MR exists. Use the result as "the base branch" in all subsequent steps.

**If GitHub:**
1. `gh pr view --json baseRefName -q .baseRefName` — if succeeds, use it
2. `gh repo view --json defaultBranchRef -q .defaultBranchRef.name` — if succeeds, use it

**If GitLab:**
1. `glab mr view -F json 2>/dev/null` and extract the `target_branch` field — if succeeds, use it
2. `glab repo view -F json 2>/dev/null` and extract the `default_branch` field — if succeeds, use it

**Git-native fallback (if unknown platform, or CLI commands fail):**
1. `git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's|refs/remotes/origin/||'`
2. If that fails: `git rev-parse --verify origin/main 2>/dev/null` → use `main`
3. If that fails: `git rev-parse --verify origin/master 2>/dev/null` → use `master`

If all fail, fall back to `main`.

Print the detected base branch name. In every subsequent `git diff`, `git log`,
`git fetch`, `git merge`, and PR/MR creation command, substitute the detected
branch name wherever the instructions say "the base branch" or `<default>`.

# /autobuild — Autonomous Plan Implementation

Plan approved? Let's build it.

/autobuild reads your approved plan file, extracts the implementation steps, and
builds everything autonomously. It works best after `/autoplan-full` or `/autoplan`
has produced a reviewed plan with architecture diagrams, test plans, and clear scope.

---

## Phase 0: Pre-flight

### Step 1: Find the plan

```bash
eval "$(~/.claude/skills/gstack/bin/gstack-slug 2>/dev/null)" && mkdir -p ~/.gstack/projects/$SLUG
BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null)
echo "Branch: $BRANCH"
```

Search for the plan file in this order:
1. Plan file referenced in CLAUDE.md or conversation context
2. Most recent plan file: `ls -t ~/.gstack/projects/$SLUG/*-plan-*.md 2>/dev/null | head -1`
3. Plan file in the current directory: `ls -t *.plan.md plan.md PLAN.md 2>/dev/null | head -1`

If no plan found, use AskUserQuestion: "No plan file found. Provide the path to your
plan file, or run `/autoplan-full` first to create one."

### Step 2: Read and validate the plan

Read the plan file completely. Verify it contains:
- [ ] Implementation scope (what to build)
- [ ] Architecture (components, data model, or system design)
- [ ] File list or component breakdown (what files to create/modify)
- [ ] Implementation Tasks section (with `- [ ] #N:` checkboxes) — if missing,
  warn but continue. Tasks are optional but improve tracking.

If the plan is missing critical sections, warn:
"This plan is missing [X]. /autobuild works best with a fully reviewed plan.
Continue anyway, or run `/autoplan-full` first?"

### Step 3: Find the test plan

Search for test plan artifact:
```bash
ls -t ~/.gstack/projects/$SLUG/*-test-plan-*.md 2>/dev/null | head -1
```

If found, read it — this guides what tests to write alongside the implementation.

### Step 4: Read supporting artifacts

- Read CLAUDE.md for project conventions (test commands, linting, frameworks)
- Read the design doc if referenced: `ls -t ~/.gstack/projects/$SLUG/*-design-*.md 2>/dev/null | head -1`
- Read DESIGN.md if it exists (design system tokens, fonts, colors)
- Read the research brief if referenced: `ls -t ~/.gstack/projects/$SLUG/*-research-*.md 2>/dev/null | head -1`

### Step 5: Pre-flight confirmation

Use AskUserQuestion:

```
## /autobuild Pre-flight

Plan: [plan file path]
Test plan: [test plan path or "none found"]
Design doc: [path or "none found"]
Branch: [branch]
Scope: [N] components, [M] files estimated

### Implementation Order
[Extract from plan: which components to build first, dependencies between them]

### Build Strategy
I'll implement in this order:
1. [Data models / types first]
2. [Core logic / services]
3. [API / routes]
4. [UI components] (if applicable)
5. [Tests alongside each component]

Ready to build?
```

Options:
- A) Build it (recommended — I'll implement everything)
- B) Build with checkpoints (pause after each major component for review)
- C) Adjust order (tell me what to change)
- D) Cancel

**If A (full autonomy):**
Proceed to Phase 1 with no interruptions. Build everything, run tests at the end.
This is the fastest path. Best for well-reviewed plans.

**If B (checkpoints):**
Proceed to Phase 1 but pause after each major component with a brief status:
"Built [component]. [N] files created, [M] tests written. Tests: [pass/fail]. Continue?"
This is safer for large plans or when the plan has ambiguity.

## Phase 1: Implementation

### Task tracking

If the plan has an `## Implementation Tasks` section with checkboxes, use it to
drive the build order and track progress:

1. Read the task list and respect dependency ordering (`depends: #N`)
2. Build tasks in dependency order — never start a task before its dependencies
3. After completing each task, check it off in the plan file using Edit:
   Change `- [ ] #N:` to `- [x] #N:` in the plan file
4. If a task is blocked by something unexpected, note it and move to the next
   unblocked task

If no task list exists, follow the implementation order from the plan as before.

### Build methodology

For each component in the implementation order (or each task if task list exists):

1. **Read existing code first.** Never write to a file you haven't read. Understand
   the patterns, conventions, imports, and style of the existing codebase.

2. **Write the implementation.** Follow the plan's architecture exactly. If the plan
   specifies a pattern, use it. If it doesn't specify, match existing codebase patterns.

3. **Write tests alongside.** If a test plan exists, follow it. If not, write tests
   for every new public function, every new API endpoint, and every new component.
   Match the existing test framework and conventions in CLAUDE.md.

4. **Run tests after each component.** Read the test command from CLAUDE.md. If tests
   fail, fix them before moving to the next component.

### Implementation rules

- **Follow the plan.** The plan was reviewed by CEO, UX, design, and eng. It encodes
  decisions that were carefully made. Do not second-guess scope, architecture, or
  approach. If something seems wrong, note it but build what was specified.

- **Match existing patterns.** Import style, naming conventions, file structure,
  error handling — match what's already in the codebase. Read 2-3 similar files
  before writing a new one.

- **No gold plating.** Build exactly what the plan says. Don't add features, don't
  refactor adjacent code, don't "improve" things outside scope. The plan defines
  the blast radius — stay inside it.

- **Tests are not optional.** Every new codepath gets a test. If the test framework
  doesn't exist yet, create it (check CLAUDE.md for preferences first).

- **Commit atomically.** After each logical unit of work (one component, one feature,
  one module), create a commit. Each commit should be independently understandable
  and revertable.

### Progress tracking

After each component (or task), output a brief status:

```
✓ [Component name] — [N] files created/modified, [M] tests
  Task: #[N] [checked off in plan file]
  Files: [list]
  Tests: [pass count]/[total] passing
```

If in checkpoint mode (Option B), use AskUserQuestion after each component.

If the plan has a task list, also show overall progress:
```
Tasks: [completed]/[total] ([percentage]%)
```

## Phase 2.5: Self-Healing Verify Loop

After all components are built but before running the full test suite, run the
verify loop to catch and fix broken workflows.

### Step 1: Scope the verification

Count how many commits this build session created:
```bash
git log --oneline -20
```

Identify the commits from THIS build session (the ones you just created in Phase 1).
Use `git diff --stat HEAD~N` where N is the number of build session commits.

### Step 2: Run /investigate-workflow logic

For each file in the build diff:
1. Trace entry points (routes, handlers, API endpoints, event listeners)
2. Follow code paths from entry to outcome
3. Categorize findings: BROKEN, INCOMPLETE, MISSING, ORPHANED, FRAGILE

Record a findings snapshot before applying any fixes.

### Step 3: Auto-fix BROKEN items

BROKEN items are definitively wrong — auto-fix without asking:
- Read the file, understand intended behavior from context
- Write the minimal fix — no refactoring
- Report each fix: what file, what changed, why

### Step 4: Present INCOMPLETE items

INCOMPLETE items may be intentional (deferred scope from the plan). Use AskUserQuestion
to present them. The user decides which to fix, which to defer.

### Step 5: Re-verify (max 3 iterations)

After fixes, re-investigate the same scope. Check for regressions:
- Compare new findings against the original snapshot
- NEW finding not in original set = REGRESSION → STOP, revert last fix, report
- Zero BROKEN remaining = done
- Same/fewer findings = continue next iteration

After 3 iterations, report any remaining items and continue to Phase 2.

## Headless Mode

/autobuild is designed to work headless. For fully autonomous execution:

```bash
claude -p "Run /autobuild on the approved plan" --allowedTools 'Bash,Read,Write,Edit,Glob,Grep'
```

In headless mode:
- Skip the pre-flight AskUserQuestion — auto-select Option A (full autonomy)
- Skip checkpoint pauses — build everything continuously
- Output the build report at the end

## Important Rules

- **Follow the plan.** The plan is the source of truth. It was reviewed. Build what it says.
- **Read before write.** Never write to a file you haven't read first. Understand the codebase.
- **Match patterns.** Import style, naming, structure — match existing code, not your preferences.
- **Tests are mandatory.** Every new codepath gets a test. No exceptions.
- **Commit atomically.** One logical unit per commit. Each commit independently revertable.
- **No gold plating.** Build exactly what was planned. Nothing more.
- **Report clearly.** After each component, show what was built and whether tests pass.
- **Never ship.** /autobuild builds — it does not ship. Run /autoaudit then /ship.
