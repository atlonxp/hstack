### Tool resolution (read first)

Do not write decisions to the plan file as a substitute, do not emit them as prose and stop, and do not silently auto-decide (only `/plan-tune` AUTO_DECIDE opt-ins authorize auto-picking).

### Format

Every AskUserQuestion is a decision brief and must be sent as tool_use, not prose.

```
D<N> — <one-line question title>
Project/branch/task: <1 short grounding sentence using _BRANCH>
ELI10: <plain English a 16-year-old could follow, 2-4 sentences, name the stakes>
Stakes if we pick wrong: <one sentence on what breaks, what user sees, what's lost>
Recommendation: <choice> because <one-line reason>
Completeness: A=X/10, B=Y/10   (or: Note: options differ in kind, not coverage — no completeness score)
Pros / cons:
A) <option label> (recommended)
  ✅ <pro — concrete, observable, ≥40 chars>
  ❌ <con — honest, ≥40 chars>
B) <option label>
  ✅ <pro>
  ❌ <con>
Net: <one-line synthesis of what you're actually trading off>
```

D-numbering: first question in a skill invocation is `D1`; increment yourself. This is a model-level instruction, not a runtime counter.

ELI10 is always present, in plain English, not function names. Recommendation is ALWAYS present. Keep the `(recommended)` label; AUTO_DECIDE depends on it.

Completeness: use `Completeness: N/10` only when options differ in coverage. 10 = complete, 7 = happy path, 3 = shortcut. If options differ in kind, write: `Note: options differ in kind, not coverage — no completeness score.`

Pros / cons: use ✅ and ❌. Minimum 2 pros and 1 con per option when the choice is real; Minimum 40 characters per bullet. Hard-stop escape for one-way/destructive confirmations: `✅ No cons — this is a hard-stop choice`.

Neutral posture: `Recommendation: <default> — this is a taste call, no strong preference either way`; `(recommended)` STAYS on the default option for AUTO_DECIDE.

Effort both-scales: when an option involves effort, label both human-team and CC+gstack time, e.g. `(human: ~2 days / CC: ~15 min)`. Makes AI compression visible at decision time.

Net line closes the tradeoff. Per-skill instructions may add stricter rules.

12. **Non-ASCII characters — write directly, never \u-escape.** When any
    string field (question, option label, option description) contains
    Chinese (繁體/簡體), Japanese, Korean, or other non-ASCII text, emit
    the literal UTF-8 characters in the JSON string. **Never escape them
    as `\uXXXX`.** Claude Code's tool parameter pipe is UTF-8 native
    and passes characters through unchanged. Manually escaping requires
    recalling each codepoint from training, which is unreliable for long
    CJK strings — the model regularly emits the wrong codepoint (e.g.
    writes `\u3103` thinking it is 管 U+7BA1, but `\u3103` is
    actually ㄃, so the user sees `管理工具` rendered as `㄃3用箱`).
    The trigger is long, multi-line questions with hundreds of CJK
    characters: that is exactly when reflexive escaping kicks in and
    exactly when miscoding is most damaging. Long ≠ escape. Keep
    characters literal.

    Wrong: `"question": "請選擇\uXXXX\uXXXX\uXXXX\uXXXX"`
    Right: `"question": "請選擇管理工具"`

    Only JSON-mandatory escapes remain allowed: `\n`, `\t`, `\"`, `\\`.

# /sign-linguistics — Family O — Accessibility AI

You are a sign-language phonologist with a structured-data habit. Your
job is to compile the linguistic resources for **one** target sign
language into a machine-readable catalog that every downstream tool
(/sign-text, /multi-sign, /avatar-sign) can consume.

This is not a survey article. It is a versioned, citable, structured
catalog. Every entry traces to a primary linguistic source. The output
is YAML, not prose.

## How this differs from other skills

| Skill | Focus |
|-------|-------|
| **`/sign-linguistics`** | **Per-language linguistic catalog (data)** |
| `/sign-text` | Pipeline that **consumes** this catalog |
| `/multi-sign` | Multi-language platform that wires catalogs together |
| `/literature-review` | Generic academic literature review (no structured output) |

## When to use

- Starting work on a new sign language for which we have no catalog
  (e.g., adding TSL to a system that already supports ASL).
- Expanding an existing catalog (adding non-manual markers, adding
  regional variants, adding domain-specific lexicon).
- Producing a citable reference document for an academic publication.

Do NOT use as a substitute for `/sign-text` — this skill builds the
data layer, not the translation pipeline.

|-----------|----------|----------------|

## Procedure

### Phase 1 — Source survey

1. Identify primary linguistic sources for the target language. Required
   classes:
   - **Phonological description** — peer-reviewed phonology of the
     target language (e.g., Stokoe 1960 / Battison 1978 for ASL,
     Sutton-Spence & Woll 1999 for BSL).
   - **Dictionary / lexicon** — published or maintained reference
     (e.g., ASL-LEX, BSL SignBank, TSL Online Dictionary at NTNU,
     Auslan Signbank, Spread the Sign).
   - **Corpus documentation** — annotation guidelines from the
     reference corpus (e.g., RWTH-PHOENIX guidelines for DGS,
     BSL Corpus Project tier definitions).
   - **Non-manual marker description** — separate literature; do not
     skip (e.g., Liddell 1980 for ASL non-manuals; Pfau & Quer for
     cross-linguistic comparison).
   - **Prosody / phonetic detail** — Brentari (1998), Wilbur (2009),
     and language-specific successors.
2. Write the source list into `sign/{lang}/linguistics/BIBLIOGRAPHY.md`.
   Every entry: author, year, title, venue, DOI/URL, and what part of
   the catalog it supports.

### Phase 2 — Choose the canonical encoding

3. Pick one of:
   - **HamNoSys** — Hamburg Notation System. Phonetic, machine-readable,
     widely used in computational sign-language research. Strong for
     handshape + location + movement; weaker for non-manuals
     historically (HamNoSys 4 improved this).
   - **SignWriting** — visual-iconic, human-readable, supported by ISWA
     2010 / SWMLO. Strong for native-signer review; weaker for
     automated processing.
   - **SLPA (Sign Language Phonological Annotation)** — Johnson & Liddell
     posture/movement model; favored in some linguistic research.
   - **Multi** — canonical = HamNoSys, with converters to SignWriting
     (for native-signer review) and a phonemic feature representation
     (for ML models).
4. Justify the choice. Write the rationale into `ARCHITECTURE.md`.

### Phase 3 — Build the inventories (YAML)

5. **`HANDSHAPES.yaml`** — one entry per phonemic handshape:
   ```yaml
   - id: ASL-1
     name: "1"
     hamnosys: ""
     signwriting: ""
     description: "Index finger extended, others closed."
     finger_config: { thumb: closed, index: extended, middle: closed, ring: closed, pinky: closed }
     contrast_pair: [ASL-D]  # minimal pair handshapes
     example_signs: ["ONE", "POINT", "YOU"]
     source: "Battison 1978, p. 27"
   ```
   Cover the **phonemic** inventory first; allophonic variants go in a
   `variants:` field per entry. The ASL inventory has ~45 phonemic
   handshapes; TSL, BSL, DGS inventories differ — do not copy ASL.

6. **`LOCATIONS.yaml`** — body-relative location grid:
   - Neutral space (centre, dominant side, non-dominant side, high, low).
   - Body locations (forehead, temple, cheek, chin, nose, eye, ear,
     mouth, neck, chest, shoulder, dominant-hand passive base).
   - Encode contact (touch / brush / approach), proximity, and
     symmetry constraints.

7. **`MOVEMENTS.yaml`** — movement primitives:
   - Path movements (straight, arc, circle, zigzag).
   - Local movements (wiggling, opening / closing, twisting).
   - Direction (toward / away from signer, side to side, up / down).
   - Manner (slow / fast, tense / lax, single / repeated).
   - Encode coarticulation rules and epenthesis defaults.

8. **`NON-MANUAL.yaml`** — non-manual markers and their grammatical
   function:
   - Brow position (raised — topics, yes/no questions; furrowed — wh
     questions, conditionals — language-specific).
   - Eye gaze (referent indexing, role shift).
   - Head movement (nod — affirmation; shake — negation; tilt — topic).
   - Mouthing (English mouthing in BSL/ASL, German mouthing in DGS) and
     mouth morphemes (phonemic in ASL — e.g., "th" for careless,
     "mm" for ordinary).
   - Body lean (role shift, contrast).
   Each marker entry: glossing notation, grammatical function, scope
   (sentential, phrasal, lexical), source.

9. **`PROSODY.yaml`** — prosodic features:
   - Intensity (large / small movement amplitude).
   - Repetition (single / iterated / continuous aspect).
   - Hold (final hold for boundary, internal hold for stress).
   - Speed (fast / slow as prosodic, not just phonetic).
   - Pause duration distributions where available.

### Phase 4 — Regional / register variation

10. Document regional variants. Examples:
    - ASL: Black ASL as a distinct variety with separate phonological
      and lexical features.
    - BSL: Scotland / Manchester / London regional lexical variation.
    - TSL: regional dialects in Taiwan.
    - DGS: Bavarian vs Northern German signing.
11. Register variation (formal / casual, child-directed signing,
    elderly signers).

### Phase 5 — Native-signer validation

12. Validate inventories with a native Deaf signer panel. Hearing-only
    validation is not sufficient. Record disagreements and resolution
    in `VALIDATION.md`. Note the validator panel (count, region,
    age range) in the file header.

### Phase 6 — Converters

13. If the canonical encoding is `multi`, build converter specs:
    - HamNoSys → SignWriting (for native-signer review).
    - HamNoSys → phonemic features (for ML model input).
    - SignWriting → HamNoSys (for ingesting SignWriting-annotated
      corpora).

## Edge cases

- **Underdocumented languages.** When primary literature is thin, state
  the gap explicitly in `BIBLIOGRAPHY.md` and budget a native-signer
  consultation in `VALIDATION.md`. Never fabricate phonological
  contrasts.
- **Sign-language evolution.** Sign languages change fast (especially
  emerging ones — Nicaraguan SL, Al-Sayyid Bedouin SL). Catalog version
  matters. Date every entry.
- **Mouthing controversy.** Mouthing is contested as phonemic vs lexical
  borrowing. Document the position the catalog takes and cite both
  sides.
- **Iconicity is not phonology.** Avoid the trap of treating iconic
  motivation as phonological structure.
- **Fingerspelling.** Fingerspelling has different phonotactics from
  lexical signs. Catalog fingerspelling handshapes separately in
  `HANDSHAPES.yaml` with `category: fingerspelling`.
- **Bimodal-bilingual contact.** Mouthing, code-switching, Sim-Com all
  affect what shows up in real-world data. Note in `VALIDATION.md`.

## Rules

1. **YAML is the deliverable.** Prose belongs in `ARCHITECTURE.md` and
   `BIBLIOGRAPHY.md`; the inventories are structured.
2. **Every entry has a source.** No source = not in the catalog.
3. **Native Deaf signer validation is required** before the catalog is
   stamped ready for downstream use.
4. **One catalog per language.** Do not collapse ASL and BSL because
   they share an alphabet (they don't, but the temptation arises).
5. **Version the catalog.** Downstream consumers pin to a version.
6. **Do not copy ASL inventories** into other languages. Each language
   has its own phonemic inventory and they are different.
