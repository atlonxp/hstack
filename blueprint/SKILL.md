## AskUserQuestion Format

### Tool resolution (read first)

"AskUserQuestion" can resolve to two tools at runtime: the **host MCP variant** (e.g. `mcp__conductor__AskUserQuestion` — appears in your tool list when the host registers it) or the **native** Claude Code tool.

**Rule:** if any `mcp__*__AskUserQuestion` variant is in your tool list, prefer it. Hosts may disable native AUQ via `--disallowedTools AskUserQuestion` (Conductor does, by default) and route through their MCP variant; calling native there silently fails. Same questions/options shape; same decision-brief format applies.

**If no AskUserQuestion variant appears in your tool list, this skill is BLOCKED.** Stop, report `BLOCKED — AskUserQuestion unavailable`, and wait for the user. Do not write decisions to the plan file as a substitute, do not emit them as prose and stop, and do not silently auto-decide (only `/plan-tune` AUTO_DECIDE opt-ins authorize auto-picking).

### Format

Every AskUserQuestion is a decision brief and must be sent as tool_use, not prose.

```
D<N> — <one-line question title>
Project/branch/task: <1 short grounding sentence using _BRANCH>
ELI10: <plain English a 16-year-old could follow, 2-4 sentences, name the stakes>
Stakes if we pick wrong: <one sentence on what breaks, what user sees, what's lost>
Recommendation: <choice> because <one-line reason>
Completeness: A=X/10, B=Y/10   (or: Note: options differ in kind, not coverage — no completeness score)
Pros / cons:
A) <option label> (recommended)
  ✅ <pro — concrete, observable, ≥40 chars>
  ❌ <con — honest, ≥40 chars>
B) <option label>
  ✅ <pro>
  ❌ <con>
Net: <one-line synthesis of what you're actually trading off>
```

D-numbering: first question in a skill invocation is `D1`; increment yourself. This is a model-level instruction, not a runtime counter.

ELI10 is always present, in plain English, not function names. Recommendation is ALWAYS present. Keep the `(recommended)` label; AUTO_DECIDE depends on it.

Completeness: use `Completeness: N/10` only when options differ in coverage. 10 = complete, 7 = happy path, 3 = shortcut. If options differ in kind, write: `Note: options differ in kind, not coverage — no completeness score.`

Pros / cons: use ✅ and ❌. Minimum 2 pros and 1 con per option when the choice is real; Minimum 40 characters per bullet. Hard-stop escape for one-way/destructive confirmations: `✅ No cons — this is a hard-stop choice`.

Neutral posture: `Recommendation: <default> — this is a taste call, no strong preference either way`; `(recommended)` STAYS on the default option for AUTO_DECIDE.

Effort both-scales: when an option involves effort, label both human-team and CC+gstack time, e.g. `(human: ~2 days / CC: ~15 min)`. Makes AI compression visible at decision time.

Net line closes the tradeoff. Per-skill instructions may add stricter rules.

12. **Non-ASCII characters — write directly, never \u-escape.** When any
    string field (question, option label, option description) contains
    Chinese (繁體/簡體), Japanese, Korean, or other non-ASCII text, emit
    the literal UTF-8 characters in the JSON string. **Never escape them
    as `\uXXXX`.** Claude Code's tool parameter pipe is UTF-8 native
    and passes characters through unchanged. Manually escaping requires
    recalling each codepoint from training, which is unreliable for long
    CJK strings — the model regularly emits the wrong codepoint (e.g.
    writes `\u3103` thinking it is 管 U+7BA1, but `\u3103` is
    actually ㄃, so the user sees `管理工具` rendered as `㄃3用箱`).
    The trigger is long, multi-line questions with hundreds of CJK
    characters: that is exactly when reflexive escaping kicks in and
    exactly when miscoding is most damaging. Long ≠ escape. Keep
    characters literal.

    Wrong: `"question": "請選擇\uXXXX\uXXXX\uXXXX\uXXXX"`
    Right: `"question": "請選擇管理工具"`

    Only JSON-mandatory escapes remain allowed: `\n`, `\t`, `\"`, `\\`.

## Model-Specific Behavioral Patch (claude)

The following nudges are tuned for the claude model family. They are
**subordinate** to skill workflow, STOP points, AskUserQuestion gates, plan-mode
safety, and /ship review gates. If a nudge below conflicts with skill instructions,
the skill wins. Treat these as preferences, not rules.

**Todo-list discipline.** When working through a multi-step plan, mark each task
complete individually as you finish it. Do not batch-complete at the end. If a task
turns out to be unnecessary, mark it skipped with a one-line reason.

**Think before heavy actions.** For complex operations (refactors, migrations,
non-trivial new features), briefly state your approach before executing. This lets
the user course-correct cheaply instead of mid-flight.

**Dedicated tools over Bash.** Prefer Read, Edit, Write, Glob, Grep over shell
equivalents (cat, sed, find, grep). The dedicated tools are cheaper and clearer.

## Voice

GStack voice: Garry-shaped product and engineering judgment, compressed for runtime.

- Lead with the point. Say what it does, why it matters, and what changes for the builder.
- Be concrete. Name files, functions, line numbers, commands, outputs, evals, and real numbers.
- Tie technical choices to user outcomes: what the real user sees, loses, waits for, or can now do.
- Be direct about quality. Bugs matter. Edge cases matter. Fix the whole thing, not the demo path.
- Sound like a builder talking to a builder, not a consultant presenting to a client.
- Never corporate, academic, PR, or hype. Avoid filler, throat-clearing, generic optimism, and founder cosplay.
- No em dashes. No AI vocabulary: delve, crucial, robust, comprehensive, nuanced, multifaceted, furthermore, moreover, additionally, pivotal, landscape, tapestry, underscore, foster, showcase, intricate, vibrant, fundamental, significant.
- The user has context you do not: domain knowledge, timing, relationships, taste. Cross-model agreement is a recommendation, not a decision. The user decides.

Good: "auth.ts:47 returns undefined when the session cookie expires. Users hit a white screen. Fix: add a null check and redirect to /login. Two lines."
Bad: "I've identified a potential issue in the authentication flow that may cause problems under certain conditions."

## Completeness Principle — Boil the Lake

AI makes completeness cheap. Recommend complete lakes (tests, edge cases, error paths); flag oceans (rewrites, multi-quarter migrations).

When options differ in coverage, include `Completeness: X/10` (10 = all edge cases, 7 = happy path, 3 = shortcut). When options differ in kind, write: `Note: options differ in kind, not coverage — no completeness score.` Do not fabricate scores.

# /blueprint — Family J — Product Transposition

Study a target — a live product, a research paper, an open-source codebase, an app,
or an API/SDK — and produce a **domain-agnostic BLUEPRINT.md** capturing its
mechanism. Output is reusable downstream by `/transpose`, `/clone-and-twist`, and
`/recipe`. Output is **not** a derivative reproduction — it is a transformative
analysis that lets you build something new in your own domain.

This is the first stop of Hall's constructive reverse-engineering pipeline:

```
  /blueprint  →  /transpose  →  /clone-and-twist  →  /recipe
   study           plan the         build the          document
   THEIRS          bridge           transposed         YOURS
```

## When to use

- You found a product (fingerspelling.xyz, an ASL pedagogy app) and want to
  understand its teaching mechanism before transposing to Thai Sign Language.
- You're studying a research paper + reference code (supertonic-v3) and want
  to extract the mechanism so you can transpose inference → fine-tuning.
- You're scouting an open-source repo to learn its architecture before building
  something adjacent in your own domain.
- You want a single durable artifact instead of scattered Notion/Obsidian/head-notes.

**Not for:** verbatim cloning (refused), studying auth-walled targets without
ownership declaration (refused), or adversarial reverse engineering on
unauthorized targets (use Family F instead).

## Inputs

| Flag | Required | Meaning |
|---|---|---|
| `<target>` | yes | URL / arXiv ID / repo URL / app store ID / OpenAPI URL |
| `--mode` | yes | One of: `product`, `paper`, `code`, `app`, `api` |
| `--focus <area>` | no | Repeatable. Areas to deepen (`--focus pedagogy --focus prosody`) |
| `--exclude <area>` | no | Repeatable. Known noise to skip (`--exclude marketing-copy`) |
| `--owned` | no | Declares you own / have authorization for an auth-walled target |
| `--slug <name>` | no | Override auto-derived study slug |

If `--mode` is missing, infer from the target shape then **AskUserQuestion** to
confirm before continuing.

## Modes

### `--mode product` (live web/SaaS product)

Studies a live product end-to-end via `/browse` (observation) and public docs
(reading). **Refuses to operate behind auth without `--owned`.** Captures:

- UX flow: the screens, the sequence, the affordances, the empty/error states.
- Behavior probes: how does the product respond to nonsense input, edge cases,
  rate limits (only on free-tier / public surfaces).
- Public docs: feature lists, changelog, blog, marketing claims (cited).
- Pricing / packaging: what tier exposes what behavior.

Extractor: `lib/study-extractors/product.ts` (planned). Uses `/browse` for
observation, `/scrape` within bounds, `markitdown` for downloaded docs.

### `--mode paper` (research paper)

Ingests one or more papers (PDF, arXiv ID, OpenReview link) via `markitdown`
and extracts:

- Architecture: model / system structure, with diagrams when available.
- Method: training procedure, inference procedure, key algorithms.
- Results + ablations: which design choices mattered, by how much.
- Data: datasets used (with provenance), preprocessing decisions.
- Open questions: what the paper didn't ablate, what's load-bearing but unproven.

Every claim cites a page/section. Hooks `huggingface-skills:hugging-face-paper-publisher`
for citation formatting.

### `--mode code` (open-source codebase)

Static + light dynamic analysis of an OSS repo. **Public repos only.** Captures:

- Entry points (CLI, library API, server routes).
- Key abstractions: classes, modules, types that carry the mechanism.
- Call graph for the critical path.
- Non-obvious design choices surfaced from comments, ADRs, commit messages.
- Test suite shape — tests reveal intended behavior the README didn't say.

Every described mechanism cites `file:line`. Refuses to operate on a repo
without a license; warns loudly on copyleft licenses (GPL, AGPL) since
downstream transposition output could inherit obligations — the user must
ack the implication via AskUserQuestion before continuing.

### `--mode app` (mobile app)

Mobile app via **public surface only**:

- App store metadata (description, screenshots, reviews surfacing UX).
- Public marketing site + docs.
- Behavioral observation against documented endpoints / web APIs the app
  uses (when present and public).
- Manifest / public configuration when available.

**Refuses decompilation of proprietary binaries.** If the user wants binary
analysis of an app they own, route them to Family F (`/binary-analysis`) and
require the appropriate scope gate.

### `--mode api` (API / SDK)

OpenAPI/SDL spec + behavioral probing of documented endpoints, surfaces:

- Resource model + verbs.
- Auth model + rate limits as documented.
- Behavioral surprises: undocumented but observable behavior, surfaced as
  **hypotheses** (not asserted facts) with replay evidence.
- Error model: how the API signals each failure class.

Probing stays within the documented surface, respects rate limits, identifies
itself via User-Agent, and **stops at any signal of unwanted load**.

## Procedure

### Step 0: Pre-flight

```bash
eval $(~/.claude/skills/gstack/bin/gstack-slug 2>/dev/null)
TARGET_SLUG="<derived-from-target-or---slug>"
mkdir -p "studies/$TARGET_SLUG/evidence" "studies/$TARGET_SLUG/notes"
```

Decide:
1. **What is the target?** Confirm URL/arXiv-ID/repo/app/API.
2. **What mode?** If user didn't pass `--mode`, infer + confirm via AskUserQuestion.
3. **Is it auth-walled or proprietary?**
   - If `product` mode and target is behind login: STOP. AskUserQuestion:
     "This target appears to require login. Do you own / have written authorization
     to study it? Answer YES only if you can produce that authorization on request."
     If the answer is anything but YES (+ `--owned`), refuse and exit with a
     clear message.
   - If `app` mode and only proprietary binary is available: refuse, suggest Family F.
   - If `code` mode and repo has no license: refuse. If GPL/AGPL: warn + AskUserQuestion
     to confirm the user understands transposition output may inherit obligations.

### Step 1: Scope the study

Use AskUserQuestion:

> What's the scope of this study?
>
> **A) Mechanism only** — extract just the core algorithm/architecture. 30 min.
> **B) Mechanism + UX** — algorithm plus the surface that users touch. 60 min.
> **C) Full blueprint** — mechanism, UX, data model, edge cases, open questions. 90+ min.
> **D) Custom** — describe the focus.

If `--focus` was passed, use it to bias the study; still ask the depth question above.

### Step 2: Dispatch to the mode extractor

Pseudo-flow per mode:

- **product**: `/browse` the public surface → screenshot key states → read public
  docs via `markitdown` → write observations to `studies/<slug>/evidence/`.
- **paper**: `markitdown <pdf-or-arxiv-url>` → extract architecture / method /
  results / data / ablations → write per-claim citations to `citations.md`.
- **code**: `git clone <repo>` to a temp dir (or use repo browser tools) →
  identify entry points → trace the critical path → record `file:line` cites.
  Never modify or commit to the studied repo.
- **app**: pull store metadata, read marketing site, observe public web/API
  surface via `/browse`. Do not decompile.
- **api**: fetch spec → probe documented endpoints respectfully → record
  request/response examples under `evidence/`.

Each extractor writes raw findings to `studies/<slug>/notes/raw-<mode>.md`
**before** synthesis. Raw notes preserve where each claim came from.

### Step 3: Synthesize the domain-agnostic blueprint

Read all raw notes and produce `studies/<slug>/BLUEPRINT.md` with this YAML
frontmatter (required by S12):

```yaml
---
target: <name + url>
mode: product | paper | code | app | api
version: 1
derived_from: []   # blueprints can chain (e.g., paper-blueprint derived_from a prior product-blueprint)
extracted_on: <ISO date>
slug: <target-slug>
focus: [list]
excluded: [list]
---
```

The body has these sections, in order:

1. **One-paragraph thesis** — what this thing does, in one sentence a smart
   outsider could understand. No jargon.
2. **Mechanism** — the core thing that makes it work. Domain-agnostic; do not
   bake in source-domain vocabulary that would block transposition.
3. **Data model** — the entities, their relationships, their lifecycles.
4. **Key algorithms** — named, described, cited. ASCII diagram if non-trivial.
5. **Edge cases (observed)** — what we saw the target handle and how.
6. **Edge cases (probed)** — what we tested explicitly; pass/fail.
7. **Non-obvious design choices** — the "why this, not the obvious thing" calls.
8. **Open questions** — what we could NOT observe; load-bearing unknowns. These
   become bridges or stubs in `/transpose`.
9. **Mechanism properties to preserve** — candidate inputs to the S13 fidelity
   contract. Each property must be observable / measurable on a transposed version.

### Step 4: Write citations and questions

- `studies/<slug>/citations.md` — every external claim with source URL, page/file:line,
  retrieval date. Required for legal-review traceability.
- `studies/<slug>/questions.md` — open questions surfaced during the study,
  tagged `[answerable]`, `[needs-experiment]`, or `[unresolvable-from-outside]`.

### Step 5: Self-check (do not skip)

Before writing, verify all of the following. If any check fails, fix before
exiting the skill — do not declare the study done.

- [ ] BLUEPRINT.md has the required YAML frontmatter (S12).
- [ ] **No verbatim copy:** no source code blocks, no marketing copy, no UI
      strings, no images, no logos. Reference + cite — do not paste.
- [ ] Mechanism is described in **domain-agnostic** terms (not "ASL" but
      "phoneme inventory"; not "Stripe" but "tokenized card object").
- [ ] Every non-obvious claim has a citation (URL, page, file:line, or
      observation-log timestamp).
- [ ] At least 3 candidate mechanism properties listed for S13 fidelity.
- [ ] `questions.md` lists at least the load-bearing unknowns.

### Step 6: Report

Print summary:

```
Study complete: studies/<slug>/
- Mode: <mode>
- Target: <url>
- BLUEPRINT.md: <line count> lines, <section count> sections
- Evidence files: <N>
- Citations: <N>
- Open questions: <N>
- Candidate mechanism properties: <N>

Next steps:
  /transpose <slug>         — plan the bridge to your target domain
  /recipe --for paper <slug> — if studying for a literature review
```

## Edge cases

- **Auth-walled target without ownership declaration.** Refuse. Print:
  `/blueprint refuses to study auth-walled targets without --owned. If you have
  authorization, re-run with --owned.`
- **Proprietary binary (mobile app only).** Refuse decompilation. Suggest `/binary-analysis`
  (Family F) with the appropriate scope gate.
- **Repo with no license / restrictive license.** Refuse / AskUserQuestion to confirm.
- **Target is behind a paywall but content is technical (paper, docs).** Treat as
  auth-walled. Require `--owned` or an explicit acknowledgement of personal
  legitimate access (e.g., institutional subscription).
- **Verbatim-copy temptation.** If you find yourself wanting to quote more than
  ~25 words consecutively from the target, stop. Paraphrase + cite instead.
- **Source code in `code` mode.** Cite `file:line`. Do NOT paste large source
  blocks into BLUEPRINT.md. Small expression-level snippets for clarity are OK
  with attribution; functions are not.
- **`app` mode with no public surface.** If the app has no public docs / store
  description / discoverable API, refuse — there is nothing you can ethically
  blueprint without binary analysis.
- **Existing study with same slug.** Bump the `version:` field; never silently
  overwrite. Old version moves to `studies/<slug>/archive/v<N>/`.
- **Citation rot.** Snapshot referenced URLs to `evidence/` so future readers
  aren't blocked by dead links.
