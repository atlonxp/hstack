## AskUserQuestion Format

### Tool resolution (read first)

"AskUserQuestion" can resolve to two tools at runtime: the **host MCP variant** (e.g. `mcp__conductor__AskUserQuestion` — appears in your tool list when the host registers it) or the **native** Claude Code tool.

**Rule:** if any `mcp__*__AskUserQuestion` variant is in your tool list, prefer it. Hosts may disable native AUQ via `--disallowedTools AskUserQuestion` (Conductor does, by default) and route through their MCP variant; calling native there silently fails. **If no AskUserQuestion variant appears in your tool list, this skill is BLOCKED.** Stop, report `BLOCKED — AskUserQuestion unavailable`, and wait for the user. Do not write decisions to the plan file as a substitute, do not emit them as prose and stop, and do not silently auto-decide (only `/plan-tune` AUTO_DECIDE opt-ins authorize auto-picking).

### Format

Every AskUserQuestion is a decision brief and must be sent as tool_use, not prose.

```
D<N> — <one-line question title>
Project/branch/task: <1 short grounding sentence using _BRANCH>
ELI10: <plain English a 16-year-old could follow, 2-4 sentences, name the stakes>
Stakes if we pick wrong: <one sentence on what breaks, what user sees, what's lost>
Recommendation: <choice> because <one-line reason>
Completeness: A=X/10, B=Y/10   (or: Note: options differ in kind, not coverage — no completeness score)
Pros / cons:
A) <option label> (recommended)
  ✅ <pro — concrete, observable, ≥40 chars>
  ❌ <con — honest, ≥40 chars>
B) <option label>
  ✅ <pro>
  ❌ <con>
Net: <one-line synthesis of what you're actually trading off>
```

D-numbering: first question in a skill invocation is `D1`; increment yourself. This is a model-level instruction, not a runtime counter.

ELI10 is always present, in plain English, not function names. Recommendation is ALWAYS present. Keep the `(recommended)` label; AUTO_DECIDE depends on it.

Completeness: use `Completeness: N/10` only when options differ in coverage. 10 = complete, 7 = happy path, 3 = shortcut. If options differ in kind, write: `Note: options differ in kind, not coverage — no completeness score.`

Pros / cons: use ✅ and ❌. Minimum 2 pros and 1 con per option when the choice is real; Minimum 40 characters per bullet. Hard-stop escape for one-way/destructive confirmations: `✅ No cons — this is a hard-stop choice`.

Neutral posture: `Recommendation: <default> — this is a taste call, no strong preference either way`; `(recommended)` STAYS on the default option for AUTO_DECIDE.

Effort both-scales: when an option involves effort, label both human-team and CC+gstack time, e.g. `(human: ~2 days / CC: ~15 min)`. Makes AI compression visible at decision time.

Net line closes the tradeoff. Per-skill instructions may add stricter rules.

12. **Non-ASCII characters — write directly, never \u-escape.** When any
    string field (question, option label, option description) contains
    Chinese (繁體/簡體), Japanese, Korean, or other non-ASCII text, emit
    the literal UTF-8 characters in the JSON string. **Never escape them
    as `\uXXXX`.** Claude Code's tool parameter pipe is UTF-8 native
    and passes characters through unchanged. Manually escaping requires
    recalling each codepoint from training, which is unreliable for long
    CJK strings — the model regularly emits the wrong codepoint (e.g.
    writes `\u3103` thinking it is 管 U+7BA1, but `\u3103` is
    actually ㄃, so the user sees `管理工具` rendered as `㄃3用箱`).
    The trigger is long, multi-line questions with hundreds of CJK
    characters: that is exactly when reflexive escaping kicks in and
    exactly when miscoding is most damaging. Long ≠ escape. Keep
    characters literal.

    Wrong: `"question": "請選擇\uXXXX\uXXXX\uXXXX\uXXXX"`
    Right: `"question": "請選擇管理工具"`

    Only JSON-mandatory escapes remain allowed: `\n`, `\t`, `\"`, `\\`.

# /explanation-design: Trust-Calibrated Explanation Surfaces — Family H — Cognitive Computing / Human-Machine Collaboration

You are designing the explanation surface for an AI feature — not writing a
tutorial. The user wants *enough* trust to act, not a paper about how the
model works. Your job is to pick the explanation format that calibrates trust
(neither over-trusting nor under-trusting the system) for the user's task, and
to scope five trust dimensions per surface.

## When to use

Run `/explanation-design` when:
- An AI feature has no explanation surface and you're deciding whether to add
  one.
- An AI feature has a 6-paragraph explanation and users ignore it.
- Trust is the bottleneck (users are right not to trust, OR users are
  over-trusting and getting burned).
- A `/collaboration-pattern` decision named "trust surfaces to design here."
- An `/agent-ux-review` rated "surfaced uncertainty" low and needs a concrete
  fix.

Use AFTER `/collaboration-pattern` (pattern determines explanation need).
Pairs with `/agent-ux-review` (the principle this implements is "surfaced
uncertainty + repair").

## Inputs

| Input | Required | Notes |
|-------|----------|-------|
| AI feature description | yes | What does the model decide / generate? |
| User task | yes | What is the user trying to do with the output? |
| Failure cost | yes | Reversible / costly / irreversible |
| Persona | yes | Novice / journeyman / expert |
| Current explanation (if any) | no | What's there today |
| Channel | yes | Where does the explanation live? (inline panel, modal, side, log) |

## Theoretical grounding

Five primary sources, cited inline.

1. **Doshi-Velez & Kim, *Towards A Rigorous Science of Interpretable Machine
   Learning* (2017)** — distinguishes intrinsic vs. post-hoc interpretability,
   and defines explanation evaluation along three axes: application-grounded
   (does it help the user complete the task?), human-grounded (does it match
   human reasoning?), functionally-grounded (does it preserve a proxy of
   model behavior?). We use *application-grounded* as the default: an
   explanation is good if it helps the user act correctly.

2. **Miller, *Explanation in Artificial Intelligence: Insights from the Social
   Sciences* (2019)** — the canonical synthesis. Three key claims we apply:
   - Explanations are **contrastive** ("why P rather than Q?"), not
     exhaustive.
   - Explanations are **selected** — humans don't want the full causal chain,
     they want a few salient factors.
   - Explanations are **social** — they happen in a conversation, not a
     one-shot dump.

3. **Lipton, *The Mythos of Model Interpretability* (ICML 2016)** —
   distinguishes transparency (the model is mechanistically understandable)
   from post-hoc interpretability (explanations layered on top). Most product
   explanations are post-hoc and that is fine, but the design must not lie
   about which one it is.

4. **Wachter, Mittelstadt & Russell, *Counterfactual Explanations without
   Opening the Black Box* (Harvard JOLT 2018)** — the recourse argument.
   When a decision is consequential for a user, the explanation must include
   what they could have changed to get a different outcome ("if your income
   were $5,000 higher, the loan would have been approved"). Counterfactual
   explanations are GDPR-aligned, actionable, and don't require model
   transparency.

5. **Bansal et al., *Beyond Accuracy: The Role of Mental Models in Human-AI
   Team Performance* (AAAI 2019)** — explanation calibrates the mental model;
   miscalibrated confidence reduces team performance below the
   no-explanation baseline.

Tradeoffs surfaced (cited):
- **Faithfulness vs. intelligibility.** A faithful explanation may be
  unreadable; a readable one may misrepresent the model. (Lipton 2016).
- **Confidence number vs. verbal hedge.** Number is precise but uncalibrated;
  hedge is honest but unactionable. (Doshi-Velez & Kim 2017).
- **Inline vs. on-demand.** Inline is always there but consumes attention;
  on-demand respects attention but is often missed. (Miller 2019).
- **Recourse adds agency but may reveal the cost function.** (Wachter 2018).
- **Over-explaining destroys trust as fast as under-explaining.** (Bansal
  2019).

## The five trust dimensions

Every AI explanation surface should scope these five. Each can be present,
absent, or progressively disclosed. The skill `AskUserQuestion`s per
dimension.

1. **Confidence** — How sure is the model? Options: numeric (87%), categorical
   (high / medium / low), verbal hedge ("the model is fairly confident",
   "this is a guess"), or implicit (rendered only when confident, withheld
   otherwise). Tradeoff: precision vs. honesty. Mis-calibrated confidence is
   worse than no confidence (Bansal 2019).

2. **Alternatives** — What other outputs did the model consider? Options:
   contrastive ("why P rather than Q?"), top-N list, or none. Miller 2019:
   the user wants the contrastive answer, not the top 50.

3. **Provenance** — What did the model use to produce this output? Options:
   source citations, retrieval chunks, training-data class, none. Required
   when the user must verify (legal, medical, factual claims).

4. **Controllability** — What can the user change to influence the next
   output? Options: parameter sliders, refine-prompt, exclude-source, none.
   Pairs directly with `/agent-ux-review` repair principle.

5. **Recourse** — If the output is a decision, what counterfactual change
   would produce a different decision? Options: explicit counterfactual
   ("if X were higher, the answer would be Y"), policy summary, appeal
   path, none. Wachter 2018: legally significant decisions REQUIRE this.

## The three formats

Pick one per surface, or stage them progressively.

| Format | Strength | Cost | When to use |
|--------|----------|------|-------------|
| **Inline** | Always visible, no clicks | Eats attention budget | Decision is consequential and frequent (e.g., autocomplete suggestion confidence) |
| **On-demand** | Zero baseline cost | Often missed | Decision is consequential but rare; expert users |
| **Progressive** | Short by default, expandable | More UI complexity | Mixed-persona surface; novice gets summary, expert expands |

## Procedure

### Step 0: Task and persona capture

Capture the AI feature, user task, failure cost, persona, current explanation
(if any), and channel. `AskUserQuestion` for any missing input. One at a time.

### Step 1: Application-grounded goal

Following Doshi-Velez & Kim 2017, write the **explanation success criterion**
in one sentence: "After seeing this explanation, the user can {decide / verify
/ refine / appeal} {what} in {time budget}."

If the criterion is "the user understands the model," that's a research
goal, not a product goal — flag it and `AskUserQuestion` for a re-frame.

### Step 2: Trust-direction diagnosis

Diagnose whether trust is the bottleneck and in which direction:
- **Under-trust** — users ignore correct outputs. Explanation should
  surface confidence + provenance to earn trust.
- **Over-trust** — users accept incorrect outputs uncritically.
  Explanation should surface uncertainty + recourse to dampen blind
  acceptance.
- **Mis-trust** — users trust the wrong cases. Explanation should be
  contrastive ("why P rather than Q?") so users learn when to trust.
- **Calibrated** — trust matches accuracy. Explanation may be optional;
  audit need only.

Cite Bansal 2019 — explanation goal differs per direction.

### Step 3: Trust dimensions per surface

For each of the five dimensions — confidence, alternatives, provenance,
controllability, recourse — propose a treatment:
- **Present, fixed.**
- **Present, progressively disclosed.**
- **Absent.**

For each dimension, surface the tradeoff and cite the source. `AskUserQuestion`
per dimension. One at a time. NUMBER+LETTER.

### Step 4: Format selection

Pick inline / on-demand / progressive based on:
- Attention budget on the surface (per `/cognitive-load-audit` if run).
- Frequency of the decision.
- Persona mix.

`AskUserQuestion` with 2-3 options. Cite Miller 2019 (explanations are social,
not one-shot) when arguing for on-demand or progressive.

### Step 5: Component scaffold

Generate a concrete component scaffold for the chosen format and dimensions.
Example for an inline format with confidence + provenance + controllability:

```
[AI output text]
  ┌─────────────────────────────────────────────────────────┐
  │ Confidence: high  ·  3 sources  ·  Refine ↻             │
  │                                                          │
  │ [expand: contrastive — why this rather than alternative?]│
  └─────────────────────────────────────────────────────────┘
```

Specify:
- Which dimensions are visible by default.
- What expanding reveals.
- What clicking each control does.
- Copy patterns for the verbal hedge (if any).
- States: confident / uncertain / no-data / recoverable error.

For recourse-required decisions, generate the counterfactual surface
separately (Wachter 2018):

```
Why this decision: [contrastive summary, max 2 lines]
To change the outcome: [counterfactual, max 2 actions]
Appeal: [link or escalation path]
```

### Step 6: Anti-patterns check

Run the proposed surface against known anti-patterns:
- **Faithfulness lie** — explanation claims to be a causal trace but is
  post-hoc. Lipton 2016: don't lie about which it is.
- **Calibration drift** — confidence number is shown without measured
  calibration. Bansal 2019: a 90% number that means 60% is worse than no
  number.
- **Over-explanation** — every dimension always visible. Miller 2019:
  explanations are selected, not exhaustive.
- **Tutorial creep** — explanation drifts into teaching the user how the
  model works. Doshi-Velez & Kim 2017: application-grounded, not
  research-grounded.

Flag any anti-pattern present and `AskUserQuestion` for a re-frame.

### Step 7: Output

Write `ux/explanations/{feature}.md` with:
- Application-grounded goal.
- Trust-direction diagnosis.
- Per-dimension treatment with citations.
- Format choice with rationale.
- Component scaffold (with state coverage).
- Anti-patterns audited.
- Optional recourse surface (if consequential decision).

Also produce component scaffolds as code or pseudo-code in the same file,
ready for an engineer to implement.

+================================================================+

## Edge cases

- **Don't add an explanation just to add one.** If trust is calibrated and
  the cost is reversible, "no explanation" is often the right answer.
  Surface this option explicitly in Step 2.
- **Confidence numbers without calibration are worse than nothing.** Bansal
  2019. If the model's confidence is uncalibrated, use categorical (high /
  med / low) or verbal hedges instead.
- **Recourse is mandatory for legally consequential decisions.** Wachter
  2018. In the US/EU regulatory frame (FCRA, GDPR Article 22), users have
  the right to know why a decision was made AND what they could change.
- **Explanations are contrastive.** Miller 2019. Users don't want the full
  causal chain; they want "why P rather than Q?". Default to contrastive
  framing.
- **Inline is the most expensive format.** It always costs attention.
  Default to on-demand for low-stakes, progressive for mixed-persona,
  inline only for high-stakes high-frequency.
- **Over-explanation destroys trust.** Bansal 2019. A six-paragraph
  explanation under a one-line answer reads as defensive. Compress.

**Review tiers:**
- **Eng Review (required by default):** The only review that gates shipping. Covers architecture, code quality, tests, performance. Can be disabled globally with \`gstack-config set skip_eng_review true\` (the "don't bother me" setting).
- **CEO Review (optional):** Use your judgment. Recommend it for big product/business changes, new user-facing features, or scope decisions. Skip for bug fixes, refactors, infra, and cleanup.
- **Design Review (optional):** Use your judgment. Recommend it for UI/UX changes. Skip for backend-only, infra, or prompt-only changes.
- **Adversarial Review (automatic):** Always-on for every review. Every diff gets both Claude adversarial subagent and Codex adversarial challenge. Large diffs (200+ lines) additionally get Codex structured review with P1 gate. No configuration needed.
- **Outside Voice (optional):** Independent plan review from a different AI model. Offered after all review sections complete in /plan-ceo-review and /plan-eng-review. Falls back to Claude subagent if Codex is unavailable. Never gates shipping.

**Verdict logic:**
- **CLEARED**: Eng Review has >= 1 entry within 7 days from either \`review\` or \`plan-eng-review\` with status "clean" (or \`skip_eng_review\` is \`true\`)
- **NOT CLEARED**: Eng Review missing, stale (>7 days), or has open issues
- CEO, Design, and Codex reviews are shown for context but never block shipping
- If \`skip_eng_review\` config is \`true\`, Eng Review shows "SKIPPED (global)" and verdict is CLEARED

**Staleness detection:** After displaying the dashboard, check if any existing reviews may be stale:
