### Format

Every AskUserQuestion is a decision brief and must be sent as tool_use, not prose.

```
D<N> — <one-line question title>
Project/branch/task: <1 short grounding sentence using _BRANCH>
ELI10: <plain English a 16-year-old could follow, 2-4 sentences, name the stakes>
Stakes if we pick wrong: <one sentence on what breaks, what user sees, what's lost>
Recommendation: <choice> because <one-line reason>
Completeness: A=X/10, B=Y/10   (or: Note: options differ in kind, not coverage — no completeness score)
Pros / cons:
A) <option label> (recommended)
  ✅ <pro — concrete, observable, ≥40 chars>
  ❌ <con — honest, ≥40 chars>
B) <option label>
  ✅ <pro>
  ❌ <con>
Net: <one-line synthesis of what you're actually trading off>
```

D-numbering: first question in a skill invocation is `D1`; increment yourself. This is a model-level instruction, not a runtime counter.

ELI10 is always present, in plain English, not function names. Recommendation is ALWAYS present. Keep the `(recommended)` label; AUTO_DECIDE depends on it.

Completeness: use `Completeness: N/10` only when options differ in coverage. 10 = complete, 7 = happy path, 3 = shortcut. If options differ in kind, write: `Note: options differ in kind, not coverage — no completeness score.`

Pros / cons: use ✅ and ❌. Minimum 2 pros and 1 con per option when the choice is real; Minimum 40 characters per bullet. Hard-stop escape for one-way/destructive confirmations: `✅ No cons — this is a hard-stop choice`.

Neutral posture: `Recommendation: <default> — this is a taste call, no strong preference either way`; `(recommended)` STAYS on the default option for AUTO_DECIDE.

Effort both-scales: when an option involves effort, label both human-team and CC+gstack time, e.g. `(human: ~2 days / CC: ~15 min)`. Makes AI compression visible at decision time.

Net line closes the tradeoff. Per-skill instructions may add stricter rules.

12. **Non-ASCII characters — write directly, never \u-escape.** When any
    string field (question, option label, option description) contains
    Chinese (繁體/簡體), Japanese, Korean, or other non-ASCII text, emit
    the literal UTF-8 characters in the JSON string. **Never escape them
    as `\uXXXX`.** Claude Code's tool parameter pipe is UTF-8 native
    and passes characters through unchanged. Manually escaping requires
    recalling each codepoint from training, which is unreliable for long
    CJK strings — the model regularly emits the wrong codepoint (e.g.
    writes `\u3103` thinking it is 管 U+7BA1, but `\u3103` is
    actually ㄃, so the user sees `管理工具` rendered as `㄃3用箱`).
    The trigger is long, multi-line questions with hundreds of CJK
    characters: that is exactly when reflexive escaping kicks in and
    exactly when miscoding is most damaging. Long ≠ escape. Keep
    characters literal.

    Wrong: `"question": "請選擇\uXXXX\uXXXX\uXXXX\uXXXX"`
    Right: `"question": "請選擇管理工具"`

    Only JSON-mandatory escapes remain allowed: `\n`, `\t`, `\"`, `\\`.

# /prosody-control — Family L — Speech & Voice AI

You are running the `/prosody-control` skill. This designs the **prosody-control
surface** for a TTS system: how the user (or the agent calling the TTS) shapes
expression, tone, accent, cadence, and emphasis. Out-of-the-box TTS sounds flat —
this skill picks the right combination of techniques and lays out the training-data
strategy to make them work.

This is a **design** skill. It produces planning documents, not trained models.
Downstream `/finetune --domain speech-tts` consumes the plan; `/voice-eval` measures
whether the surface is doing what it claims.

The fingerspelling.xyz → TSL case is the canonical motivating example: Thai-accented
English with teaching-cadence prosody (slow, repeated, exaggerated syllable boundaries
for a learner). No off-the-shelf model does this. The prosody-control surface is what
makes it buildable.

---

## When to use

- TTS sounds flat / robotic and needs expression.
- Need accent control (e.g., American → Thai-accented English) without re-training a
  whole base model.
- Need emotion control (happy / serious / urgent) on a fixed voice.
- Building a teaching / explainer voice with deliberate cadence (Hall's TSL case).
- Multi-speaker mixing (blend two voices, e.g., 70% Hall + 30% another speaker).
- Per-sentence style needed in long-form narration (e.g., audiobook chapters with
  different moods).

Do NOT use for:
- Cloning the voice itself → `/voice-clone`.
- Picking the base TTS architecture → `/tts-design` first.
- Eval-only work → `/voice-eval`.

### Step 2: Pick the control surface composition

The prosody-control surface is composed of 1-5 layers. Pick the combination per
use case.

| Layer | What it does | When to use | Cost |
|---|---|---|---|
| **Reference-audio conditioning** | "Style transfer from this clip" | Have clean style examples; want zero-shot transfer | Low |
| **Global Style Tokens (GST)** | Learn N latent style vectors from training data | Want categorical control ("happy/sad/teaching") with smooth interpolation | Medium (training) |
| **SSML extensions** | Explicit `<prosody rate="slow" pitch="+15%">` + custom `<style>` / `<accent>` / `<emphasis>` tags | Want author-controlled, deterministic output | Low (parsing only) |
| **IPA + tone marks** | Phoneme-level control with tone diacritics (Mandarin, Thai, Vietnamese) | Tonal languages, accent control, pronunciation fixes | Medium (g2p infra) |
| **Per-sentence style embeddings** | Compute style vector per utterance from text + tags | Long-form with shifting mood (audiobooks, explainers) | Medium |
| **Multi-speaker mixing** | Linear-interpolate speaker embeddings (e.g., 0.7·SpeakerA + 0.3·SpeakerB) | Voice blending, persona dial | Low |
| **Natural-language style prompts** | "Read this in a calm teacher voice" → embedded by an aux text encoder | Modern flow-matching models (F5-TTS, some 2024 SOTA) | High (extra model) |

AskUserQuestion (for each layer): include or skip? Show the cost + use case fit.

- One layer per section.
- For each: which technique, which library / config flag, what the user-facing

### Step 3: SSML / tag vocabulary

If SSML or custom tags are in the surface, write
`$PLAN_ROOT/{slug}/PROSODY/SSML-EXTENSIONS.md` defining:

- **W3C SSML 1.1 baseline:** `<break>`, `<prosody rate pitch volume>`, `<emphasis>`,
  `<say-as>`, `<phoneme>`, `<sub>`.
- **hstack extensions (custom):**
  - `<style name="...">` — references a GST token or a style embedding by name.
  - `<accent code="en-th">` — pushes the synthesizer toward an accent embedding.
  - `<cadence preset="teaching">` — preset that combines rate, pitch range, pause

  - `<repeat count="2" mode="exaggerate">` — for teaching cadence (repeat with

support it (graceful degradation, not silent drop).

### Step 4: Style data plan

Write `$PLAN_ROOT/{slug}/PROSODY/STYLE-DATA-PLAN.md`. This is the augmentation
strategy to make the layers actually work.

| Layer | Data requirement | Augmentation |
|---|---|---|
| Reference-audio conditioning | 10-50 clean utterances per style (already in target voice) | None — keep clean |
| GST | 50-500 utterances per style, balanced across speakers | Speed perturbation (±10%), pitch shift (±2 semitones) — careful not to wash out style |
| SSML extensions | Test corpus where each tag is exercised + annotated ground truth | Synthetic — generate matched pairs by varying tags on same text |
| IPA + tone marks | Phoneme-aligned corpus; for tonal langs, tone-balanced | Forced alignment via Montreal Forced Aligner |
| Per-sentence style embeddings | Long-form corpus with sentence-level style annotation | Crowdsource style tags OR derive from emotion classifier |
| Multi-speaker mixing | Multi-speaker corpus with consistent recording conditions | Volume + RIR normalization |
| Natural-language style prompts | Paired (prompt, audio) corpus | Hard to augment; data quality > quantity |

For each row, also write:
- **Existing public datasets:** ESD (emotion), IEMOCAP (emotion), CSS10 (multi-lang),

- **Phase 4:** train SSML tag parsers (if used) — these are usually rule-based,

- **Style classifier accuracy:** train a small classifier on style tags (or use

## Edge cases

- **Multilingual code-switching** (e.g., Thai-accented English with mid-sentence
  Thai phrases): XTTSv2 is the cleanest base. The control surface needs explicit
  `<lang>` boundaries OR a language-detection preprocessor. Without it, the
  synthesizer will mispronounce code-switched segments.
- **Conflicting style requests** (e.g., "fast newsreader cadence" + "intimate
  whispered tone" simultaneously): surface the conflict. Style mixing in latent
  space is allowed but produces unpredictable acoustics — recommend the user
  pick one dominant style.
- **Accent + tonal-language interaction** (Thai → Mandarin accent): tonal language
  accents are harder than non-tonal because tone-realization patterns differ.
  Flag this; recommend a tone-aware g2p + native-speaker eval panel.
- **Style data scarcity:** if user wants 8 styles but has data for 3, recommend
  training the 3 + using natural-language prompts for the others (only on
  prompt-capable backbones like F5-TTS or 2024+ models).
- **SSML tag the synthesizer doesn't support:** parser must degrade gracefully
  (drop tag, log warning) — never raise to the user.
- **Hall's TSL case** (Thai-accented English, teaching cadence, fingerspelling →
  TSL voice): full layer stack — reference-audio conditioning + GST for cadence
  presets + SSML extensions for `<repeat mode="exaggerate">` + IPA for explicit
  pronunciation of fingerspelled letters. Recommend StyleTTS2 base, custom
  recording session for ~30 minutes of teaching-cadence Thai-accented English.
- **Watermark interaction:** if `/voice-clone` produced the base voice with
  AudioSeal, verify the prosody training doesn't degrade watermark recovery.
  Cite the test in EVAL-PERCEPT.md.

## Important Rules

- **Never design prosody before base architecture is locked.** This skill requires
  a `/tts-design` slug. If none exists, refuse and route the user there.
- **Always cite a degraded-fallback behavior for each SSML tag.** Synthesizers
  evolve; tags must never crash the pipeline.
- **Style requests that contradict physically** (e.g., "whisper" + "shout") get a
  surfaced warning, not a silent best-effort blend.
- **For cloned voices: verify watermark robustness post-prosody-training.**
  Prosody fine-tuning can subtly corrupt watermark signal.
- **Eval-percept is mandatory.** Don't ship a prosody surface that hasn't been
  panel-rated.
- **The teaching-cadence use case (Hall's TSL fingerspelling work) is a first-class
  pattern, not an edge case.** Treat it as the canonical example throughout.
