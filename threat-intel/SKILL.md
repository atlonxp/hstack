## AskUserQuestion Format

### Tool resolution (read first)

"AskUserQuestion" can resolve to two tools at runtime: the **host MCP variant** (e.g. `mcp__conductor__AskUserQuestion` — appears in your tool list when the host registers it) or the **native** Claude Code tool.

**Rule:** if any `mcp__*__AskUserQuestion` variant is in your tool list, prefer it. **If no AskUserQuestion variant appears in your tool list, this skill is BLOCKED.** Stop, report `BLOCKED — AskUserQuestion unavailable`, and wait for the user. Do not write decisions to the plan file as a substitute, do not emit them as prose and stop, and do not silently auto-decide (only `/plan-tune` AUTO_DECIDE opt-ins authorize auto-picking).

### Format

Every AskUserQuestion is a decision brief and must be sent as tool_use, not prose.

```
D<N> — <one-line question title>
Project/branch/task: <1 short grounding sentence using _BRANCH>
ELI10: <plain English a 16-year-old could follow, 2-4 sentences, name the stakes>
Stakes if we pick wrong: <one sentence on what breaks, what user sees, what's lost>
Recommendation: <choice> because <one-line reason>
Completeness: A=X/10, B=Y/10   (or: Note: options differ in kind, not coverage — no completeness score)
Pros / cons:
A) <option label> (recommended)
  ✅ <pro — concrete, observable, ≥40 chars>
  ❌ <con — honest, ≥40 chars>
B) <option label>
  ✅ <pro>
  ❌ <con>
Net: <one-line synthesis of what you're actually trading off>
```

D-numbering: first question in a skill invocation is `D1`; increment yourself. This is a model-level instruction, not a runtime counter.

ELI10 is always present, in plain English, not function names. Recommendation is ALWAYS present. Keep the `(recommended)` label; AUTO_DECIDE depends on it.

Completeness: use `Completeness: N/10` only when options differ in coverage. 10 = complete, 7 = happy path, 3 = shortcut. If options differ in kind, write: `Note: options differ in kind, not coverage — no completeness score.`

Pros / cons: use ✅ and ❌. Minimum 2 pros and 1 con per option when the choice is real; Minimum 40 characters per bullet. Hard-stop escape for one-way/destructive confirmations: `✅ No cons — this is a hard-stop choice`.

Neutral posture: `Recommendation: <default> — this is a taste call, no strong preference either way`; `(recommended)` STAYS on the default option for AUTO_DECIDE.

Effort both-scales: when an option involves effort, label both human-team and CC+gstack time, e.g. `(human: ~2 days / CC: ~15 min)`. Makes AI compression visible at decision time.

Net line closes the tradeoff. Per-skill instructions may add stricter rules.

12. **Non-ASCII characters — write directly, never \u-escape.** When any
    string field (question, option label, option description) contains
    Chinese (繁體/簡體), Japanese, Korean, or other non-ASCII text, emit
    the literal UTF-8 characters in the JSON string. **Never escape them
    as `\uXXXX`.** Claude Code's tool parameter pipe is UTF-8 native
    and passes characters through unchanged. Manually escaping requires
    recalling each codepoint from training, which is unreliable for long
    CJK strings — the model regularly emits the wrong codepoint (e.g.
    writes `\u3103` thinking it is 管 U+7BA1, but `\u3103` is
    actually ㄃, so the user sees `管理工具` rendered as `㄃3用箱`).
    The trigger is long, multi-line questions with hundreds of CJK
    characters: that is exactly when reflexive escaping kicks in and
    exactly when miscoding is most damaging. Long ≠ escape. Keep
    characters literal.

    Wrong: `"question": "請選擇\uXXXX\uXXXX\uXXXX\uXXXX"`
    Right: `"question": "請選擇管理工具"`

    Only JSON-mandatory escapes remain allowed: `\n`, `\t`, `\"`, `\\`.

## Procedure

### 1. Ingest

For each enabled feed, fetch the prior window's events.

| Feed type | Client |
|---|---|
| MISP | `pymisp` — `PyMISP.search('events', timestamp='1d')` |
| OpenCTI | `pycti` — GraphQL query for indicators / reports |
| STIX 2.1 bundle | `stix2` parser — read TAXII collection or local JSON |
| OSINT RSS | RSS reader + per-source HTML parser |
| Vendor CSV / JSON | per-vendor parser |

Cache raw fetches at `~/.gstack/intel/raw/{feed-id}/{date}.json` for
replayability.

### 2. Normalize to S15 IoC schema

Each indicator becomes a row matching the S15 schema:

```json
{
  "ioc_type": "ip|domain|url|hash_sha256|hash_md5|email|cert_sha1|yara|sigma",
  "value": "<the indicator>",
  "confidence": "low|medium|high",
  "tags": ["campaign:lapsus", "attack.t1190", "actor:apt29"],
  "sightings": [{"feed": "misp-circl", "ts": "..."}],
  "first_seen": "<earliest sighting>",
  "last_seen": "<latest sighting>",
  "relevant_to": ["aws", "typescript"]  // matches stack_relevance fields
}
```

### 3. Relevance filter

For each normalized indicator, compute a relevance score against
`stack_relevance`:

- **Match on tags / context** — does the campaign / actor / technique
  target the user's tech stack (TypeScript, Next.js, AWS, Okta) or vendor
  list (Stripe, Slack)?
- **Match on geography** — does the campaign target `countries_of_interest`?
- **Match on data type** — does the campaign target the kind of data the
  project handles (PII / PHI / financial)?

Indicators with zero matches drop. Indicators with one or more matches land
in the store with `relevant_to` populated.

### 4. Dedup + enrich

For each candidate:

- Lookup against the existing `intel/iocs.jsonl` store; if the indicator
  is already known, merge `sightings` and update `last_seen`. Don't
  duplicate.
- Enrich confidence: if the same indicator appears in 3+ feeds, bump
  confidence one tier (medium → high).
- Cross-check confidence floor — drop indicators below the floor.

Write the merged store atomically (write to `.tmp`, fsync, rename).

### 5. Daily / weekly briefing

Render `intel/feeds/{YYYY-MM-DD}/BRIEFING.md`:

```markdown
# Threat Intel Briefing — {YYYY-MM-DD}

## Top relevant items (last 24h)

### 1. <campaign or actor name>
- Stack relevance: aws, typescript (from supply-chain compromise of npm pkg X)
- Confidence: high (4 feeds)
- IoCs added: 12 (3 domains, 9 hashes)
- Recommended actions:
  - /detect-engineering for T1195.002 with these hashes
  - /threat-hunt: did anyone in our org install pkg X in the last 90d?
  - /log-analyze CI logs for the listed domains

### 2. ...

## Stats
- Feeds polled: N
- Total indicators ingested: N
- After relevance filter: N
- New (not previously seen): N
- Updated (sightings added): N
- Dropped below confidence floor: N

## Feed health
- MISP-CIRCL: ok (last successful poll: 2026-05-16T03:00:00Z)
- OpenCTI-internal: STALE (last success: 2026-05-12) — investigate
```

A briefing with no high-relevance items still ships — the absence is the
signal.

## Edge cases

- **Feed authentication failure** — never echo or write the API key. Mark
  the feed STALE in the briefing and log the auth error category
  (401 / 403 / network) to `~/.gstack/intel/errors.log`. Do not include
  error bodies that might contain the key.
- **Massive feed bursts** — campaigns can dump 10k+ IoCs in a single
  bundle. Process in batches; relevance filter first, then dedup, to keep
  memory bounded.
- **Conflicting confidence between feeds** — when two feeds disagree, take
  the higher confidence but record both `sightings` so the disagreement
  surfaces in audit.
- **Legal-share-only feeds** — some vendor feeds prohibit redistribution.
  The store stays local; don't commit `intel/iocs.jsonl` if any feed in
  the source set is marked `share_policy: private` in `feeds.yaml`.
- **Indicator decay** — indicators older than 180 days with `last_seen`
  not refreshed should be marked `decayed: true` (not deleted — historical
  hunts still cite them) and excluded from active detection-engineering.
- **Out-of-scope campaigns** — even if relevance score is zero, log a
  weekly count of "out-of-scope CTI volume" so we know what we're filtering
  out. The filter being wrong is also a risk.
